# Req-Elicitor Stage 1~4 설계 분석 메모

## 요청 요약
- 월드모델 학습용 물리기반 임무환경에 대해 `req-elicitor` Stage 1~4 산출물을 작성한다.
- 단일 머신 기준으로 동작하되, GPU가 있는 PC에서 하드웨어 상태에 따라 자원을 동적으로 할당한다.
- 시뮬레이션 시간 배속은 기준 시나리오에서 활성화되는 전체 운동모델 스택 기준으로 실시간 대비 최소 `60x`를 목표로 한다.
- 월드모델 성공 판정 우선순위는 아래와 같다.
  1. 1-step / n-step prediction error
  2. latent rollout consistency
  3. 정책 학습 수렴
- 관측은 `상태벡터 + 단순 지형/위협 feature`를 기본으로 하고, 추후 이미지/가상 센서 확장을 가능하게 한다.
- 정밀 공력 계수 튜닝은 가능 여부를 검토해 가능한 범위는 이번 단계에 포함한다.
- 포토리얼 렌더링, HIL/SIL, 유인기 통제권 전환, 실측 데이터 보정은 이번 단계에서 제외한다.

## 핵심 판단

### 1. 하드웨어 적응형 rollout 수
- 고정 `병렬 rollout 수`와 `초당 env step 수`를 요구사항으로 박는 것은 이번 단계에 적합하지 않다.
- 이유는 단일 머신 기반이라도 CPU 코어 수, GPU 메모리, RAM에 따라 최적 병렬도가 크게 달라지기 때문이다.
- 따라서 요구사항은 아래처럼 잡는 것이 타당하다.
  - 시스템은 시작 시 하드웨어 자원을 감지하고 안전한 병렬 rollout 수를 자동 산정해야 한다.
  - 기준 시나리오에서 활성화되는 전체 운동모델 스택 기준으로 `60x` 시간 배속을 만족해야 한다.
  - 자동 산정 결과와 실제 throughput을 로그/리포트로 남겨야 한다.

### 2. 정밀 공력 계수 튜닝 포함 범위
- 결론: "완전한 정밀 튜닝"은 이번 단계에서 위험하지만, "튜닝 가능한 구조와 초기 보정"은 포함 가능하다.
- JSBSim 등 오픈 모델은 초기 계수 seed, 구조 참조, 회귀 기준으로 활용 가능하다.
- 가능한 범위:
  - 6-DoF 강체 모델
  - 파라미터화된 공력 계수 테이블 또는 함수
  - 계수 파일/설정 기반 오버라이드
  - JSBSim 등 공개 모델 기반 초기 계수 import 또는 비교 리포트
  - 정적/동적 sanity check와 회귀 시험
  - 수치 안정성 및 기대 기동 범위 검증
- 어려운 범위:
  - 실비행 데이터 기반 정밀 식별
  - 전영역 고신뢰 공력 검증
  - 실제 기체 수준의 고정밀 계수 보장
- 따라서 요구사항 문서에는 "정밀 공력 계수 튜닝 지원"을 넣되, 오픈 모델 활용은 bootstrap 수단으로 정의하고 정확도 보장은 `Unvalidated` 가정으로 남겨야 한다.

### 3. 관측 범위
- 기본 관측은 저차원 구조화 feature로 제한하는 것이 적절하다.
- 이유:
  - 월드모델 1차 검증에서는 렌더링 비용보다 동역학 학습 안정성이 우선이다.
  - 이후 이미지/가상 센서 확장은 인터페이스 호환성으로 대비하면 된다.

### 4. 성공 판정 우선순위 반영
- 평가 요구는 아래 우선순위로 설계한다.
  - prediction error 우선
  - latent rollout consistency 차순위
  - policy convergence는 후속 검증
- 이는 REQ와 evidence 구조에도 반영해야 한다.

## 산출물 작성 방침
- `problem_statement`: 학습 검증 중심 프로젝트로 정의
- `clarification_log`: 사용자 답변 반영, 남은 미확정점은 최소화
- `assumptions_and_constraints`: 60x, 단일 머신, 적응형 자원 할당, 관측 범위, 제외 범위 명시
- `requirements`: 기능/비기능 요구를 분리하고 모든 acceptance criteria를 수치화

## 판단 근거
- 사용자는 realism 최대화보다 월드모델 학습 검증을 우선한다고 명시했다.
- 동시에 6-DoF와 snapshot/control reinjection을 요구했으므로, 동역학 일관성과 branch rollout 능력이 핵심이다.
- 병렬 rollout 성능은 중요하지만 하드웨어 의존성이 크므로 절대 개수보다 자동 산정과 기준 시나리오 성능 보장이 요구사항으로 더 적절하다.
